CryptoPayment Gateway

Кастодиальный шлюз приёма криптовалютных платежей. Ваши пользователи отправляют крипту на персональные адреса — система автоматически обнаруживает поступление, подтверждает и перемещает средства на мастер-кошелёк, уведомляя вас через вебхук на каждом шаге.

Поддерживаемые сети
Ethereum BNB Chain TRON Solana
Поддерживаемые токены
ETH BNB TRX SOL USDT USDC
Ключевые возможности
  • Постоянные адреса per-пользователь
  • Автодетект входящих переводов
  • Автосвип на мастер-кошелёк
  • Вебхуки с подписью HMAC
  • Управление через REST API
Архитектурная модель
  • HD-кошельки (BIP44)
  • Зашифрованные приватные ключи
  • Отдельный воркер-процесс
  • Очереди через BullMQ / Redis
  • PostgreSQL для состояния

Быстрый старт

Полный сценарий — от получения ключа до первого платежа — в четырёх шагах.

1. Получите API ключ

Ключ создаётся в панели администратора или через Admin API. Ключ выглядит как cpg_<64 hex символа>.

2. Запросите депозитный адрес для пользователя

POST /api/v1/deposits/address
X-Api-Key: cpg_your_key_here
Content-Type: application/json

{
  "userId":  "user_42",
  "network": "ETH",
  "token":   "ETH"
}

// Ответ
{
  "data": {
    "userId":    "user_42",
    "network":   "ETH",
    "token":     "ETH",
    "address":   "0xABCD...1234",
    "createdAt": "2026-05-15T10:00:00.000Z"
  }
}
Адрес постоянный. Повторный запрос с тем же userId + network + token вернёт тот же адрес — никаких дублей.

3. Покажите адрес пользователю

Отобразите QR-код или строку адреса. Пользователь отправляет любую сумму.

4. Получите вебхук при зачислении

// POST на ваш webhookUrl при обнаружении перевода
{
  "event":     "payment.detected",
  "paymentId": "pay_uuid",
  "userId":    "user_42",
  "network":   "ETH",
  "token":     "ETH",
  "amount":    "1000000000000000",  // в базовых единицах (wei)
  "status":    "DETECTED",
  "txHash":    "0xdeadbeef...",
  "timestamp": "2026-05-15T10:01:30.000Z"
}

Архитектура

Модель постоянных кошельков

Каждый ваш пользователь (userId) получает уникальный адрес на каждой сети для каждого токена. Адрес не меняется со временем — пользователь всегда отправляет на один и тот же адрес.

userId="user_42" + network=ETH + token=ETH  →  0xABCD...1234
userId="user_42" + network=ETH + token=USDT →  0xEF01...5678  (другой адрес)
userId="user_99" + network=ETH + token=ETH  →  0x9876...DCBA  (другой пользователь)

Как работает детект

Воркер-процесс постоянно сканирует чейн для каждой зарегистрированной пары адрес + токен. При обнаружении входящего перевода система автоматически создаёт запись платежа и двигается по жизненному циклу.

DETECTED
Перевод в блоке
→
CONFIRMED
N подтверждений
→
SWEEPING
Перевод на мастер
→
COMPLETED
Вебхук отправлен

Деривация кошельков (HD BIP44)

Все адреса выводятся детерминированно из одной мастер-мнемоники. Приватные ключи хранятся в БД в зашифрованном виде (AES-256-GCM). Мнемоника и ключ шифрования передаются только через переменные окружения.

СетьПуть деривацииАлгоритм
ETH / BSCm/44'/60'/0'/1/{index}secp256k1
TRONm/44'/195'/0'/1/{index}secp256k1 + Base58Check
Solanam/44'/501'/{index}'/1'ed25519

Инфраструктура

API процесс
Fastify HTTP сервер. Принимает запросы, управляет кошельками и платежами, отдаёт статусы.
Worker процесс
Block watcher (сканер чейнов), Sweep worker (перевод на мастер), Webhook worker (доставка событий).

Аутентификация

Все клиентские эндпоинты требуют API ключ в заголовке X-Api-Key. Административные эндпоинты — заголовок X-Admin-Secret.

# Клиентские запросы
X-Api-Key: cpg_ae487d5e9816c9566ed4f7a0...

# Административные запросы
X-Admin-Secret: your_admin_secret
Никогда не передавайте API ключ на фронтенд. Все запросы к Gateway должны идти с вашего сервера (backend-to-backend).
ЗаголовокПрименяется кОписание
X-Api-KeyКлиентские роутыИдентифицирует клиента (мерчанта)
X-Admin-Secret/admin/*Доступ к панели управления

Жизненный цикл платежа

Платёж создаётся автоматически в момент обнаружения входящего перевода на адрес пользователя. Вручную создавать его не нужно.

СтатусОписаниеВебхук
DETECTED Перевод найден в блоке, ожидаем подтверждения payment.detected
CONFIRMED Набрано достаточно подтверждений payment.confirmed
SWEEPING Идёт перевод средств на мастер-кошелёк —
COMPLETED Средства получены, комиссия удержана payment.completed
FAILED Ошибка при свипе (нет газа, RPC недоступен) payment.failed

СетьПодтвержденийВремя ожидания
ETH12~2.5 мин
BSC15~45 сек
TRON19~1 мин
Solana1~1 сек

Значения задаются через ETH_CONFIRMATIONS, BSC_CONFIRMATIONS и т.д. в .env.

Вебхуки

Gateway делает POST запрос на ваш webhookUrl при каждом изменении статуса. Если ваш сервер вернул не 2xx — Gateway повторит доставку с экспоненциальной задержкой (до 5 попыток).

Настройка webhook URL

Задаётся через панель администратора при создании/редактировании клиента. Должен быть доступен публично по HTTPS.

Структура события

POST https://your-server.com/webhook
Content-Type: application/json
X-Webhook-Signature: sha256=a3f9b2c1...   // HMAC-SHA256 если задан webhookSecret

{
  "event":     "payment.confirmed",
  "paymentId": "550e8400-...",
  "userId":    "12345",
  "network":   "ETH",
  "token":     "USDT",
  "amount":    "50000000",
  "status":    "CONFIRMED",
  "txHash":    "0xdeadbeef...",
  "timestamp": "2026-05-15T10:03:00.000Z"
}

Типы вебхук-событий

СобытиеКогдаПоля
payment.detected Перевод обнаружен в блоке paymentId, userId, network, token, amount, txHash
payment.confirmed Набрано N подтверждений paymentId, userId, network, token, amount, txHash
payment.completed Средства получены мастер-кошельком paymentId, userId, network, token, amount, txHash (sweep tx)
payment.failed Ошибка при свипе paymentId, userId, network, token
withdrawal.completed Вывод исполнен withdrawalId, network, token, amount, txHash
withdrawal.failed Ошибка вывода withdrawalId, network, token

Безопасность вебхуков

Если у клиента задан webhookSecret, каждый запрос подписывается. Проверяйте подпись перед обработкой.

// Node.js — проверка подписи
const crypto = require('crypto');

function verifyWebhook(body, signature, secret) {
  const expected = 'sha256=' +
    crypto.createHmac('sha256', secret)
          .update(body)          // raw body bytes
          .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(signature)
  );
}

// Express
app.post('/webhook', express.raw({ type: '*/*' }), (req, res) => {
  const sig = req.headers['x-webhook-signature'];
  if (!verifyWebhook(req.body, sig, process.env.WEBHOOK_SECRET)) {
    return res.status(401).send('Invalid signature');
  }
  const event = JSON.parse(req.body);
  // обрабатываем event...
  res.status(200).send('ok');
});
Всегда проверяйте подпись. Без этого любой желающий может отправить фейковый вебхук на ваш сервер.

Идемпотентность

Используйте paymentId / withdrawalId как идемпотентный ключ. Один и тот же событие может прийти несколько раз (ретраи, network issues) — ваш обработчик должен быть устойчив к дублям.
